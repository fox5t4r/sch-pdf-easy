## Troubleshooting Case Study

문제 해결 기록은 단순히 에러와 해결 명령어만 남기지 않고, STAR 구조로 정리합니다.

### STAR 기록 기준

| 항목        | 작성 내용                                |
| --------- | ------------------------------------ |
| Situation | 문제가 발생한 상황, 배경, 서비스의 어느 부분에서 발생했는지   |
| Task      | 이 문제로 인한 사용자 경험 저하, 해결 목표, 지켜야 했던 지표 |
| Action    | 원인 분석을 위해 세운 가설, 검증 과정, 시도한 해결 방법    |
| Result    | 해결 결과, 개선 효과, 배운 점, 남은 한계와 trade-off |

---

## Case 1. 강의자료실 0KB 다운로드 문제

### Situation

강의자료실에서 일부 파일을 다운로드할 때 파일은 생성되지만 실제 내용이 없는 0KB 파일로 저장되는 문제가 발생했습니다.
강의콘텐츠에서는 정상적으로 다운로드되었지만, 강의자료실 페이지에서만 문제가 재현되었습니다.

### Task

단순히 에러 메시지를 없애는 것이 아니라, 사용자가 강의자료실에서도 실제로 열 수 있는 파일을 안정적으로 다운로드할 수 있게 만드는 것이 목표였습니다.

해결해야 할 과제는 다음과 같았습니다.

* 강의자료실과 강의콘텐츠의 파일 제공 방식 차이 확인
* 0KB 파일이 생성되는 원인 분석
* 실제 다운로드 가능한 파일 URL 또는 식별자 확보
* 기존 강의콘텐츠 다운로드 흐름은 깨지지 않도록 유지

### Action

먼저 강의콘텐츠와 강의자료실에서 파일 정보가 추출되는 방식을 비교했습니다.

초기 가설은 DOM에 표시된 링크만으로 다운로드 URL을 만들 수 있다는 것이었지만, 강의자료실에서는 화면에 보이는 정보와 실제 다운로드에 필요한 내부 식별자가 다른 경우가 있었습니다.

이후 다음 순서로 원인을 좁혔습니다.

1. 강의콘텐츠와 강의자료실의 DOM 구조 비교
2. 파일명, contentId, 다운로드 URL 매핑 여부 확인
3. React/Redux 내부 상태에서 파일 메타데이터가 존재하는지 확인
4. DOM 스캔만으로 부족한 경우 내부 상태 탐색과 fallback 로직을 함께 사용
5. 다운로드 요청 전 URL과 파일명을 검증하도록 처리

그 결과 강의자료실은 단순 DOM 스캔만으로는 실제 다운로드 가능한 정보를 안정적으로 얻기 어렵다고 판단했습니다.
따라서 내부 상태 기반 추출을 우선하고, 실패 시 DOM fallback을 사용하는 방식으로 구조를 변경했습니다.

### Result

강의자료실에서 발생하던 0KB 다운로드 문제를 줄이고, 실제 다운로드 가능한 파일 정보를 더 안정적으로 추출할 수 있게 되었습니다.

이 과정에서 배운 점은 다음과 같습니다.

* 화면에 보이는 DOM 정보가 실제 다운로드에 필요한 데이터와 항상 일치하지 않는다.
* 크롬 확장 프로그램에서는 Content Script의 isolated world 제약 때문에 페이지 내부 상태 접근 방식을 별도로 설계해야 한다.
* 하나의 추출 방식에 의존하기보다 내부 상태 탐색과 DOM fallback을 함께 두는 것이 LMS 구조 변경에 더 강하다.

남은 한계도 있습니다.

* LMS 프론트엔드 구조가 바뀌면 내부 상태 탐색 로직도 수정이 필요할 수 있습니다.
* fallback 로직을 늘리면 안정성은 올라가지만 코드 복잡도도 증가합니다.
